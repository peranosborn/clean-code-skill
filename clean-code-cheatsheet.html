Clean Code Cheat Sheet

Based on Robert C. Martin's Clean Code philosophy
Aa

Meaningful Names

1 Reveal Intent

If you have to check the implementation to understand the name, the name failed.

2 Don't Lie

If a name promises something it doesn't deliver, it erodes trust and causes bugs.

3 Read Like English

Your code is read by humans, so make it sound like human language.

4 Make Names Searchable

Single-letter variables and magic numbers are impossible to grep for.

5 Skip Type Encoding

Let your tools handle the types while you focus on logic.

6 Tell a Story

Every function should be transparently obvious and lead the reader naturally to the next.

// Bad const d = getDogs(); const allDogs = dogs.filter(d => d.active); // Good const activeDogs = getDogsForUser(); const activeDogs = dogs.filter(dog => dog.isActive);
fn

Functions

1 Do One Thing

A function should DO something or ANSWER something, but not both. (Command-query separation.)

2 Keep Args Low

Ideal: 0–1. Acceptable: 2. Three+: use an options object. Low arg count reduces mental weight.

3 One Abstraction Level

Mixing high-level and low-level is always confusing and obscures intent.

4 Bury the Switch

Keep your high-level logic clean — hide switch statements and complex conditionals behind helpers.

5 No Side Effects

A side effect is when a function modifies state outside its own scope unexpectedly.

6 Exceptions > Error Codes

Use try/catch instead of error codes. Separate error handling from business logic.

7 DRY

Don't repeat yourself. Duplicated logic means duplicated bugs.

// Bad — does two things function logMealAndUpdate(id, items) { // ...50 lines of DB + UI + calcs } // Good — reads like a story function logMeal(id, items) { const log = createMealLog(id); const entries = buildItems(log.id, items); await saveItems(entries); }

Comments

1 Why, Not What

Code tells you how. Comments tell you why. If the code already says it, the comment is noise.

2 Comments Rot

Code evolves, comments may not. A stale comment is worse than no comment.

3 Warnings Are Good

Comments that protect code from well-meaning mistakes are valuable. Use them for gotchas.

4 Don't Apologise

When comments apologise for structure, the structure needs to change.

5 Git Handles History

Commented-out code, attributions, changelogs — git handles all of it. Delete dead code.

6 Clarify, Don't Confuse

If your team can't read it instantly, rewrite it. Comments should communicate, not confuse.

7 Rename > Comment

Stop commenting what your code does. Rename it, extract it, and let it explain itself.

// Bad — restates the obvious // get the user's dogs const dogs = getUserDogs(); // Good — explains WHY // Multiply first to avoid float drift const grams = (kg * 1000 * pct) / 100;

Formatting

1 Newspaper Order

Highest-level function at the top. Every definition below its caller. Read top-to-bottom.

2 Declare Near Usage

Declare variables close to where they're used. Proximity signals relationship.

3 Team Rules Win

Consistency beats preference. Let the team decide, then enforce with Prettier + ESLint.

4 Simplest Rules, Biggest Impact

The simplest formatting rules do the most heavy lifting. Don't overcomplicate it.

5 Vertical Space = Separation

Use blank lines to separate concepts. Related lines stay close. Unrelated code gets breathing room.

6 Group Properties

Group class/component properties in one place. Don't scatter related declarations across a file.

// Newspaper order export function Dashboard() { // high-level orchestration } function DailyOverview() { ... } function RatioChart() { ... } function QuickLogButton() { ... }

Do

  • ✓ Write it messy, then refine it
  • ✓ Make every function tell a story
  • ✓ Delete dead code — git has it
  • ✓ Use early returns to flatten nesting
  • ✓ Extract helpers until code reads like prose
  • ✓ Enforce style with automated tools
  • ✓ Comment warnings and constraints

Don't

  • ✗ Leave your first draft as final code
  • ✗ Mix abstraction levels in a function
  • ✗ Comment out code — delete it
  • ✗ Use magic numbers or single letters
  • ✗ Write functions that do AND answer
  • ✗ Apologise in comments — fix the code
  • ✗ Encode types into variable names

Quick Tests

  • ? Can you name what this function does in one phrase?
  • ? Would a new team member understand this variable?
  • ? Can you delete this comment and still understand the code?
  • ? Is every function under 20 lines?
  • ? Does each function sit at one abstraction level?
  • ? Could you search for this name and find it?
